# 新版脚本附件发送与默认入口设计

## 目标

新版脚本工作台采用类似 Codex 的附件交互：选择 DOCX 只把文件加入待发送区，用户点击发送后才执行网络请求。同时允许不带 DOCX 的纯文字首发，避免把场景生成绑定到文档输入。

脚本项目默认进入新版工作台，旧版保留为显式回退入口。

## 交互设计

- 智能问答区保留 DOCX 选择按钮。
- 选择文件后，在输入区上方显示一个紧凑的附件条目，包含文件图标、文件名、大小和移除按钮。
- 第一版仅保留一个待发送 DOCX；重新选择文件会替换当前待发送附件。
- 选择或移除附件都不调用上传、Session 或 Agent 接口。
- 发送按钮以非空文字为必要条件。支持纯文字以及文字加 DOCX，不支持没有生成目标的空文字发送。
- 发送过程中禁用输入、附件选择、附件移除和发送按钮，避免重复提交。
- 请求成功后清空文字和待发送附件；请求失败时保留二者，供用户修改或重试。

## 发送数据流

### 首次纯文字发送

1. 创建 Agent Session，请求体严格发送空 JSON 对象 `{}`。
2. 在前端 Session store 中打开新 Session，并建立 SSE 连接。
3. 向该 Session 发送文字目标。

### 首次文字加 DOCX

1. 通过底座文件 API 上传待发送 DOCX。
2. 创建 Agent Session，请求体严格发送 `{}`。
3. 将上传后的 `fileId` 附加到新 Session。
4. 在前端 Session store 中打开新 Session，并建立 SSE 连接。
5. 向该 Session 发送文字目标。

### 后续发送

- 没有待发送附件时，直接向当前 Session 发送文字。
- 有待发送附件时，先上传并附加到当前 Session，再发送文字。

## 请求校验修复

当前 Agent 的 `POST /sessions` 使用严格空对象 schema 校验请求体。前端没有发送 body，Fastify 将其解析为 `undefined`，导致 `Request validation failed`。

修复方式是在 `createAgentSession()` 中显式发送 `json: {}`。这与 Agent 已有 API 契约保持一致，不放宽服务端校验。

## 路由设计

- 新建脚本成功后进入 `/new-script?projectId=<id>`。
- 从首页打开已有脚本时直接进入 `/new-script?projectId=<id>`，移除版本选择弹窗。
- 新版工作台顶部提供“退回旧版”按钮，进入 `/script?projectId=<id>`。
- `/script` 旧版路由继续保留，不迁移数据，不改变已有保存接口。

## 错误处理

- 文件类型仍由文件选择器限制为 DOCX，并保留服务端文件身份校验。
- 上传、创建 Session、附加或发送任一步失败，统一显示可见错误，保留输入和待发送附件。
- 如果上传成功但后续步骤失败，重试会再次上传文件；本次范围不引入跨请求上传缓存或孤立文件清理机制。

## 测试范围

- Agent 客户端测试证明创建 Session 会发送 `{}` 和 JSON Content-Type。
- 新版工作台集成测试证明选择附件不发请求，点击发送后按顺序上传、创建、附加和发送。
- 集成测试证明纯文字首发不调用文件上传或附加接口。
- 集成测试证明失败时附件与文字保留，成功时清空。
- 首页测试证明新建和打开脚本默认进入新版。
- 新版工作台测试证明“退回旧版”保留 `projectId` 并导航到旧路由。

## 非目标

- 多附件上传。
- 非 DOCX 附件。
- 删除已上传但未挂载成功的孤立文件。
- 修改旧版脚本编辑器或场景生成契约。
